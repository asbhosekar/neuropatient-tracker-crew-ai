Critical & High Priority Issues

Critical XSS Vulnerability via Unsanitized LLM Output
app.py : lines ~769, ~911
LLM-generated message.content is injected directly into HTML using unsafe_allow_html=True without any sanitization. A manipulated prompt or adversarial LLM output could contain <script> tags or other malicious HTML, enabling cross-site scripting attacks. User-supplied clinical summary text also flows through agents and is rendered unsanitized.
Fix: Use html.escape() on all message.content before embedding in HTML strings, or use Streamlit's native st.markdown() without unsafe_allow_html=True for agent output.
High Deprecated datetime.utcnow() Used Throughout
src/models/schemas.py : lines 74, 75, 126, 144, 164, 196 | src/agents/report_generator.py : line 321
datetime.utcnow() has been deprecated since Python 3.12. It returns a naive datetime with no timezone info, leading to ambiguity in clinical records. The audit logger already uses datetime.now(timezone.utc) correctly, showing inconsistency across the codebase.
Fix: Global find-and-replace with datetime.now(timezone.utc). Critical for clinical records where timestamp precision and unambiguity is legally relevant.
High Telemetry Tracks Zero Tokens (Nonfunctional)
src/logging/telemetry.py : lines 361-362
The track_llm_call decorator always logs prompt_tokens=0 and completion_tokens=0, making the entire cost tracking system nonfunctional. Additionally, it always uses settings.OPENAI_MODEL regardless of whether a local LLM is in use.
Fix: Extract actual token usage from AutoGen's response metadata or the OpenAI client's usage response field. Use conditional model name based on provider.
High Singleton Pattern Prevents Testing
src/logging/audit_logger.py : line 179 | src/logging/telemetry.py : line 111
Both ClinicalAuditLogger and RuntimeTelemetry use __new__-based singletons with no reset mechanism. Tests cannot get clean instances, the _initialized check is not thread-safe, and there is no way to reconfigure logging at runtime.
Fix: Add a @classmethod reset() for testing. Use threading.Lock to make initialization thread-safe.
High VitalSigns Model Has No Validation Constraints
src/models/schemas.py : lines 86-93
Unlike NeurologicalAssessment which has ge/le constraints, VitalSigns accepts any integer/float values. A temperature of -500 or a heart rate of 99999 would pass Pydantic validation. The QA Validator agent has the correct ranges in its prompt but they are not enforced at the data model level.
Fix: Add Field(ge=..., le=...) constraints matching clinical ranges. Add a model_validator to enforce systolic > diastolic.

Medium & Low Priority Issues

Medium PHI Leaked in Log Previews
src/logging/audit_logger.py : lines 378-384 | src/orchestrator.py : line 161
Agent content previews log up to 500 chars of potentially unredacted patient data. The orchestrator's task_summary also logs up to 200 chars of the task containing clinical data. While the system has PHI hashing for specific fields, these preview logs bypass that protection.
Medium No Authentication on Streamlit UI
app.py (entire file)
Any user with network access can run analyses on patient data. No authentication, authorization, or session management is implemented. For a HIPAA-compliant clinical system, this is a significant gap.
Medium No Encryption at Rest for Log Files
src/logging/audit_logger.py
Log files containing PHI hashes and clinical content previews are stored as plaintext on disk with no encryption at rest. HIPAA technical safeguards recommend encryption for PHI.
Low Dead Code: BaseAgent.create_agent() and BackendDeveloperAgent
src/agents/base_agent.py | src/agents/backend_developer.py
BaseAgent.create_agent() and _default_llm_config() are never called -- the orchestrator creates agents directly. BackendDeveloperAgent is imported in __init__.py but never used anywhere. Its generate_crud_template() returns a placeholder string.
Low Phantom Dependencies Inflating Install
pyproject.toml
FastAPI, uvicorn, SQLAlchemy, and OpenTelemetry packages are listed as required dependencies but have no actual implementation using them. They should be moved to optional dependency groups.
Low validate_data_model() is a Stub
src/agents/clinical_architect.py : lines 153-158
ClinicalArchitectAgent.validate_data_model() always returns {{"valid": True}}, providing false assurance of data model correctness.

Architecture Assessment

BaseAgent.create_agent() is disconnected from actual usage -- the orchestrator never calls it, instead instantiating agents only to extract .system_message then creating AssistantAgent directly.

BackendDeveloperAgent exists and is exported in __init__.py but is never used in the orchestrator or anywhere else. Its generate_crud_template() returns a placeholder string.

FastAPI, uvicorn, and SQLAlchemy are listed as required dependencies in pyproject.toml but there is no API server, no database models, and no endpoints implemented.

ClinicalArchitectAgent.validate_data_model() always returns {{"valid": True}} -- a stub that provides false assurance of data model correctness without performing any actual validation.

Model pricing data in telemetry.py is hardcoded and labeled "as of late 2024" -- already outdated. Prices should be externalized or made configurable.

Strengths

Outstanding Prompt Engineering (9.5/10)

Each agent has a detailed, structured system prompt with step-by-step reasoning, clinically accurate reference tables, few-shot examples with realistic data, self-review checklists, and explicit safety guardrails. Significantly above average quality.

HIPAA-Aware Logging Architecture (8/10)

PHI hashing via SHA256, separate log streams (audit, PHI access, agent conversations), log rotation with 365-day audit and 7-year PHI retention policies, structured JSON formatting with correlation IDs.

Rich Pydantic Domain Models (7.5/10)

Well-designed clinical data structures: condition-specific assessment fields, medication records with temporal tracking, prognosis analysis with confidence scoring, and proper enum usage for categorical fields.

Clinically Accurate Reference Data (8/10)

Agent helper methods contain accurate clinical data: red flag symptoms, first-line treatments with dosages, assessment score ranges, and condition-specific progression benchmarks.

Clean Project Structure (7.5/10)

Follows Python packaging conventions with proper __init__.py files, clear module boundaries, and a sensible pyproject.toml configuration.
